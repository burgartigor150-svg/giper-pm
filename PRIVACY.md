# PRIVACY.md

Это не маркетинговый документ. Это рабочий код-этики, который реализуется
в системе. Если что-то противоречит этому файлу — фиксим систему, не файл.

## Базовые принципы

1. **Согласие явное и гранулярное.** Каждый тип трекинга — отдельный чекбокс.
   Согласие хранится в БД с timestamp и IP. Пустое согласие = ничего не трекается.

2. **Прозрачность для сотрудника.** Сотрудник видит **всё**, что система знает
   о нём, на странице `/me/data`: raw activity, timeline, скриншоты, отчёты,
   которые отправлялись о нём кому-либо.

3. **Pause-режим.** В трее всегда есть кнопка «Я не на работе». При нажатии
   агент полностью замолкает: ни окна, ни idle, ни скрины. До следующего
   ручного включения.

4. **Минимизация данных.** Не собираем больше, чем нужно для конкретной фичи.
   Никаких keylogger, mic, camera, clipboard, network sniffing.

5. **Право удалить.** Любой сотрудник может в любой момент удалить любой
   свой скриншот. Любые свои Activity-записи. Без объяснения причин.

6. **Retention.** Скриншоты — 30 дней, потом cron удаляет физически.
   Activity raw — 90 дней, потом остаются только агрегаты в TimeEntry.
   TimeEntry — постоянно (это бизнес-данные).

## Что трекаем — и под каким согласием

| Тип данных                   | Согласие             | Где собирается   |
|------------------------------|----------------------|------------------|
| Активное окно (app + title)  | WINDOW_TRACKING      | desktop-agent    |
| Активная вкладка браузера    | WINDOW_TRACKING      | browser-ext      |
| Idle time                    | IDLE_DETECTION       | desktop-agent    |
| Скриншот                     | SCREENSHOTS          | desktop-agent    |
| Линковка commits → tasks     | GIT_ACTIVITY         | webhooks         |
| Календарные встречи          | CALENDAR_SYNC        | OAuth Google     |

Если согласия `WINDOW_TRACKING` нет — агент не пишет ничего, ни одной записи.
Просто отправляет heartbeat (для статуса «онлайн»).

## Что НЕ трекаем никогда

- Содержимое окон (только title!)
- Содержимое HTTP-запросов из браузера
- Контент Slack/email/мессенджеров
- Текст набираемый с клавиатуры
- Микрофон / камера
- Личные файлы вне рабочих папок
- Геолокация (только timezone из системы)
- Network traffic
- Список установленных программ

Если в окне есть приватная информация (банковский кабинет, личная переписка)
и это попало в title — сотрудник может удалить эту запись через `/me/data`,
а ещё лучше — нажать «Я не на работе» когда занимается личным.

## Кто что видит

| Роль     | Свои данные | Команда: задачи | Команда: time | Команда: activity | Команда: скрины |
|----------|-------------|-----------------|---------------|-------------------|-----------------|
| MEMBER   | всё         | да              | агрегаты      | нет               | нет             |
| PM       | всё         | да              | детально      | агрегаты          | нет             |
| ADMIN    | всё         | да              | детально      | агрегаты          | по запросу*     |
| VIEWER   | свои        | да (read-only)  | свои          | нет               | нет             |

\* Скриншоты доступны только сотруднику и его прямому PM-у. Admin видит,
   только если сотрудник дал отдельное согласие на review (например, в
   ходе спорной ситуации).

## Юридическая сторона

giper.fm — российская компания, сотрудники в РФ + удалёнщики.

**Локализация хранения**: Postgres хостится в зоне, подходящей сотруднику
(либо РФ — Yandex Cloud / Selectel, либо ЕС — Hetzner).

**Документы для сотрудников** (генерируются автоматически на /me/data):
- Согласие на обработку ПДн (152-ФЗ)
- Согласие на трекинг активности (отдельно по каждому типу)
- Уведомление о том, что данные используются для расчёта рабочего времени

Без подписи этих документов (электронная — кнопка с записью в БД) —
автотрекинг не включается.

**GDPR-режим** для сотрудников из ЕС (Frankfurt и т.п.):
- Right to access — есть, через `/me/data`
- Right to erasure — есть, кнопка «удалить все мои данные»
- Right to portability — экспорт всего своего в JSON

## Скриншоты — отдельно

Эта функция самая чувствительная. Поэтому:

1. **Default OFF.** Включается только если сотрудник явно поставил галочку.
2. **Превью с задержкой.** После захвата — 30 секунд показывается превью
   в трее. Кнопка «удалить и не отправлять». После 30с уходит в S3.
3. **Blur по умолчанию.** Если включён — всё кроме активного окна размывается.
4. **30 дней retention.** Cron удаляет физически. Не помечает как deleted —
   реально стирает из S3 + БД.
5. **Аудит просмотра.** Каждое открытие скриншота PM-ом логируется в
   AuditLog. Сотрудник видит «PM Иван открыл ваш скриншот от 12.05 14:30».

## Что делать при инциденте

Если сотрудник пожаловался, что данные собирались без согласия:
1. Немедленно проверить `UserConsent` — что было granted, когда, с какого IP
2. Проверить `Activity` за период — были ли записи при отсутствии согласия
3. Если да — это баг в коде, фиксить и удалять записи
4. Уведомить сотрудника о результатах расследования
5. Если систематическое нарушение — отключить трекинг для всех до фикса

## Чек-лист для нового кода

Прежде чем мерджить любой PR, который собирает новые данные:
- [ ] Какое согласие нужно для этого? Если нет подходящего — добавь новый `ConsentType`
- [ ] Что показываем сотруднику на `/me/data` про эти данные?
- [ ] Какой retention? Кто удаляет?
- [ ] Кто из ролей видит? Обновлён ли матрикс выше?
- [ ] Записывается ли просмотр в AuditLog?
